## Challenges & Learnings

### 1. ~50 code review patches while implementation logs said "clean"

Three of five stories reported "Debug Log: None — implementation followed story spec exactly" yet had 5–11 real patches each. The implementation agent was checking *does it run and do tests pass?* — not *is it correct?* Those are different things.

**Most systemic bugs found in review:**

- **Silent `catch { }` blocks (2.1, 2.5):** Swallowed `JsonException` with no logging. Same pattern appeared twice in different stories.
- **No DB transaction for Flat + Tariff creation (2.4 P2):** Two separate `SaveChangesAsync` calls. If the Tariff save failed, a Flat row was committed and the user was permanently stuck in onboarding. Critical correctness bug, not caught by tests.
- **EF Core SpikeThreshold sentinel conflict (2.1):** `decimal` CLR default `0m` equals EF sentinel → DB default `2.0` was never applied. Documented in Dev Notes but not baked into implementation.
- **Optimistic update cache key mismatch (2.5 P1):** `usePatchFlat.onMutate` spread `body.name` onto the `['settings']` cache, but the type field was `flatName`. The optimistic update was a no-op for the entire story.
- **Hardcoded strings in i18n-required components (2.2, 2.3, 2.5):** Despite the Epic 1 UX self-check action item, untranslated strings reached review in three separate stories.

**Root cause:** No structured self-review pass between implementation and marking a story as "ready for review."

### 2. Azure SWA Managed API does not support .NET 10

**What happened:** SWA Free tier only supports Managed API, which does not support .NET 10. The CI pipeline went green (it only builds and packages), but at runtime the frontend received 404 errors because SWA had no valid linked backend. A relatively quick analysis identified the cause. Fix: upgrade to SWA Standard tier (9 €/month), which allows linking an external Azure Functions resource directly.

**Root cause:** Two converging failures:
1. The architecture decision to use .NET 10 was made without verifying the SWA Managed API runtime support matrix.
2. Epic 1 action item #4 (Winston: update DoD for infra/config stories to require a deployed verification step) was not completed. Had it been in place, Story 2.1 would have had a deployment gate and the conflict would have surfaced before the story was marked done.

**Note on cost:** Standard tier at 9 €/month is the correct tier for production use anyway — Free tier's auth block limitation was already encountered in Epic 1. The SKU change was overdue; the timing was just wrong.

### 3. Deferred items accumulating without a triage strategy (40+)

Items were deferred story-by-story with the label "pre-existing pattern" — but some of those patterns were being copied into new files each story, which multiplies the debt.

**Bucket A — Multiplying patterns (address before Epic 3 starts):**
- The locale dropdown custom component (`aria-haspopup`, keyboard nav, touch dismiss all deferred) was hand-rolled three times across `OnboardingIntro`, `OnboardingFlatName`, and `LocaleSettings`. Each copy carries the same deferred accessibility items. Without extraction, Epic 3 will add a fourth copy.
- `parseLocaleNumber` is copy-pasted in `OnboardingContract.tsx` and `FlatBaselineEdit.tsx`. Story 3.4 (Enter Reading kWh input) will need it again.

**Bucket B — Data integrity (verify before Epic 3):**
- `CompleteOnboardingFunction` — verify flat name is trimmed before DB insert (W8 from 2.4). `PatchFlatFunction` trims it; the onboarding endpoint may not.

**Bucket C — Genuinely deferrable (carry to Epic 3 retro):**
- `GetUserId()` null guard — pre-existing, auth guaranteed by SWA Easy Auth
- InMemory FK constraint limitations — documented project-wide
- Locale rerender flash on rollback — only visible on network error
- Visual state test gaps (tile deselection, isPending) — low correctness risk

### 4. Test placeholder pattern (Story 2.4)

The Story 2.4 test spec included scaffold tests with `expect(true).toBe(true)` as placeholders for complex interactions. Several of these were not replaced with real assertions in the final implementation. Test coverage gaps for preset tile visual deselection (AC3), non-auto-select invariant (AC4), and isPending loading state (AC16) were all deferred as a result.

*Backend — check before marking review:*
- [ ] Any two related entities being saved? → single `SaveChangesAsync` after both `.Add()` calls, or explicit transaction
- [ ] Any `catch { }` block? → must name the exception type and log at Warning level
- [ ] Any `GetValue<T>()` on `JsonNode`? → guarded or wrapped in try/catch
- [ ] Any decimal entity default? → verify EF sentinel is not `0m`; set explicitly in constructor or `HasSentinel`

*Frontend — check before marking review:*
- [ ] Any new UI text? → grep for the raw string; if it appears in JSX without `t()`, it's a bug
- [ ] Any optimistic update? → verify property name in patch body matches the exact field name in the cached type
- [ ] Any new mutation creating two resources? → verify no double-call path from parent + child
- [ ] Any locale detection? → use `.startsWith('de')` not `=== 'de-DE'`

## Key Lessons for Future Epics

1. **"Tests pass" ≠ "behaviour is correct."** A self-review pass against known failure patterns is a required step before moving to code review — not a nice-to-have.
2. **Azure service runtime support must be verified at architecture time**, not discovered in production. Any new Azure resource adoption requires a runtime support matrix check in the ADR.
3. **Deferred items need a triage policy.** When a pattern is deferred for the third time ("pre-existing"), it's no longer deferred — it's accumulated debt with a multiplier. Extract before it copies again.
4. **Test scaffolds are not tests.** Placeholder assertions (`expect(true).toBe(true)`) that reach production test files are bugs in the testing strategy. Epic 3's most complex logic (KpiCalculator, TariffResolver) demands real tests, not scaffolding.
5. **DoD action items are blockers, not backlog.** Epic 1's action item #4 was carried forward, not completed. The cost was a production deployment failure. Action items from this retro are prerequisites for Epic 3 story creation.
