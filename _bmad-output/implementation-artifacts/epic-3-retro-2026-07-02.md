## Challenges & Learnings

### 1. "Looks easy, is actually complex" — the epic's dominant pattern

Two stories needed a second review round because the domain was more subtle than the acceptance criteria implied:

- **Story 3.2** ("compute a daily average cost") required restructuring the entire response shape (`DashboardSummary` → nested `CostSummary?`) after review found the original design silently underestimated cost whenever some reading intervals had no active tariff. The acceptance criteria themselves had to be corrected after the fact to match the new shape.
- **Story 3.5** ("highlight spike days on a chart") needed round 2 because round 1's fixes (timezone conversion, a `threshold <= 0m` guard, UTC weekday labels) didn't have their own regression tests and weren't fully validated the first time.

### 2. Empty/zero-data states are the most consistently missed case

Story 3.3's cold-open gradient bug and Story 3.6's missing "no readings yet" copy (fixed in today's code review) are the same category of oversight, two stories apart: happy-path-with-data gets designed and tested; the zero-data path is an afterthought both times.

### 3. A three-times-deferred bug finally happened — and got fixed at the root today

`parseLocaleNumber`'s de-DE decimal-separator bug was flagged in the Epic 2 retro, extracted into a shared utility as prep work (correctly), then deferred again in 3.4 as "not introduced by this story." In Story 3.6's review it stopped being theoretical: pre-filling an existing decimal value into an edit field produced an ambiguous dot-decimal string that the DE parser misread, risking silent data corruption (e.g., `120.5` → `1205`). **Resolved during this retrospective**: added `formatNumberForInput()` as the correct inverse of `parseLocaleNumber`, migrated every pre-fill call site (`ReadingHistorySheet.tsx`, and `FlatBaselineEdit.tsx` — which had the identical latent bug, never triggered) to use it.

### 4. A working fix wasn't systematized, so it had to be rediscovered

Story 3.4 solved rapid-double-click duplicate submission with a `submittingRef` guard — but only in its own file. Story 3.6 needed the identical fix today because nothing carried the pattern forward as a reusable hook. **Resolved during this retrospective**: extracted `useSubmitGuard()` to `client/src/lib/`, adopted by both `EnterReadingSheet.tsx` and `ReadingHistorySheet.tsx`.

### 5. Self-review checklist worked — for the bugs it was aimed at

Epic 2's retro produced a self-review checklist targeting four specific recurring bug patterns (silent `catch {}`, missing DB transactions, optimistic-update cache key mismatches, hardcoded i18n strings). None of these four reappeared in Epic 3. But total patch volume stayed roughly flat (~40 vs. Epic 2's ~50) — Epic 3's complexity (KPI math, animation race conditions, timezone/spike logic, locale parsing) surfaced a *different* class of bug: genuine business-logic correctness issues rather than checklist-catchable code smells. The checklist is working; it just can't be the only defense.

## Resolved Live During This Retrospective

Ralf asked to close out three recurring/at-risk deferred items before Epic 4 rather than carry them forward again:

1. **Tariff `EffectiveDate` tie-break inconsistency** — `TariffResolver.ResolveAsync` and `KpiCalculator.ResolveTariff` used different tie-break rules for two tariffs sharing an `EffectiveDate`, with no DB constraint preventing the collision. Safe while only onboarding could write a tariff (guaranteed unique); Epic 4's tariff CRUD removes that guarantee on day one. **Fix**: `.IsUnique()` added to `IX_Tariffs_FlatId_EffectiveDate` (migration `MakeTariffEffectiveDateUnique`) — the tie can no longer occur. Closes this item plus two older deferred entries (Epic 2 Story 2.4's W5, Epic 3 Story 3.2's tie-break note).
2. **Double-submit guard extraction** — see Challenge #4. New `client/src/lib/useSubmitGuard.ts`, adopted by `EnterReadingSheet.tsx` and `ReadingHistorySheet.tsx`. Available for Epic 4's `TariffForm` without reinvention.
3. **Numeric upper bounds** — closes Epic 2 Story 2.4's W3, open since 2026-06-29. Team-judgment thresholds (not derived from a formal requirement): `AnnualKwhBaseline < 20000` kWh, `PricePerKwh < 10` €/kWh, `MonthlyBaseFee < 1000` €, enforced in `OnboardingValidator.cs` and (for the baseline) `PatchFlatValidator.cs`. Meter-reading `KwhValue` deliberately excluded — it's a cumulative counter, not a period value, with no obvious sane ceiling.

All three verified: 90/90 backend tests, 106/106 frontend tests, clean build, clean lint.

## Key Lessons for Future Epics

1. **"Tests pass" and "implementation followed spec" don't guarantee correctness.** Two of six stories needed a second review round despite clean first-pass completion notes — the domain (cost math, timezone logic) was more subtle than the acceptance criteria captured.
2. **A deferred item that recurs across stories is a signal, not noise.** `parseLocaleNumber`'s bug was waved off as low-risk three times before it actually caused a defect — the fourth deferral would have been the one that shipped, since Epic 4 adds the exact trigger condition (pre-filled locale-sensitive fields) twice more.
3. **A working fix should be extracted immediately, not just documented.** The double-submit guard was solved once (3.4) and had to be re-solved reactively two stories later (3.6) because nothing turned it into a shared utility.
4. **Zero-data states deserve an explicit checklist line.** Two separate stories (3.3, 3.6) missed the empty-state case in the same way. Worth adding "does this component handle zero/empty data?" to the self-review checklist.
5. **When a recurring deferred item touches a field the next epic will reuse, resolve it before that epic starts — don't defer it a third time.** This is exactly what happened with the tariff tie-break and the numeric bounds today.
