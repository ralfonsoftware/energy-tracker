# Story 6.4: Gap Detection, Interpolation & Main Meter Reconciliation

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want gaps in my smart plug timeline to be automatically filled by interpolation with a notification, and total attributed consumption to always be reconciled against my main meter,
so that my decomposition data is complete and never over-counts my actual usage.

## Acceptance Criteria

1. **Given** `InterpolationEngine.cs` processing a plug's daily timeline after parsing, **when** missing dates are detected within the plug's covered period (first to last date in the import), **then** gap date ranges are recorded for notification; missing days are filled by linear interpolation between surrounding anchor values, capped at the 7-day per-day average preceding the gap; filled rows are inserted into `SmartPlugDailyData` with `IsInterpolated = true`; zero-valued rows are **not** treated as gaps â€” only absent dates trigger interpolation (FR-26).

2. **Given** dates outside the plug's covered period (before the first export date or after the last), **when** gap detection runs, **then** those periods are not treated as gaps â€” gap detection applies only within the covered range of the export.

3. **Given** `InterpolationEngine` detects one or more gaps, **when** `ImportJob` is completed, **then** a `gapNotifications` payload is stored on the `ImportJob` listing affected `plugId` and date ranges; `GET /api/v1/flats/{flatId}/imports/{jobId}` returns this payload for frontend display: "Gap detected: {date range}. Missing days have been interpolated."

4. **Given** `ReconciliationEngine.cs` running after all plugs for a period are processed, **when** it computes the period total, **then** `Residual = MainMeterTotal âˆ’ sum(SmartPlugDailyData.KwhValue for all plugs in the period)`; the invariant `attributed + Residual = MainMeterTotal` holds within Â±0.1 kWh for clean periods and within Â±1.0 kWh for periods containing any interpolated values (FR-27); if attributed kWh exceeds the main meter total, `ImportJob.Status = Failed` with `ErrorCategory = ProcessingFailed`.

5. **Given** `InterpolationEngineTests.cs` and `ReconciliationEngineTests.cs`, **when** run, **then** interpolation tests cover: gap within covered period is filled; 7-day cap is applied; zero-value days not flagged; periods outside covered range unaffected. Reconciliation tests cover: attributed + residual = meter total within tolerance for clean and interpolated periods; over-attribution triggers failure.

6. **Given** a gap found during story creation â€” no AC above (nor `epic-6-smart-plug-import-device-registry.md`) specifies **what period** `ReconciliationEngine` reconciles or **when** it runs, and this architecture has no "all plugs for a flat are done importing" signal â€” imports are one file/plugId per `ImportJob`, each processed independently by its own blob-trigger invocation â€” **when** `ProcessImportFunction` finishes interpolating gaps for the plugId just processed, **then**: `ReconciliationEngine.ReconcileAsync` is invoked exactly once per import job, using that plugId's full covered date range (min-to-max `Date` across **all** of that plugId's `SmartPlugDailyData` rows after interpolation, not just the rows touched by this specific file) as "the period"; attributed kWh sums `SmartPlugDailyData.KwhValue` across **all** `PlugId`s for the flat within that date range (not only the plugId just imported â€” a later reconciliation naturally re-checks earlier plugs' data too); if the just-processed plugId has zero `SmartPlugDailyData` rows at all after parsing (e.g. an empty/garbage file), reconciliation is skipped entirely â€” there is no period to reconcile; if `MeterReadings` do not fully bracket the period (fewer than 2 readings for the flat, or coverage does not strictly span the period â€” see Dev Notes), reconciliation is **skipped**, not failed â€” an incomplete meter-reading history on the user's side must never be misreported as an over-attribution error on the smart-plug side.

## Dev Notes

### Why this story needs no schema change (unlike Stories 6.2 and 6.3)

Both prior stories in this epic found and closed a schema gap (`PlugId` in 6.2, `OriginalFileName` in 6.3) by reading the actual as-merged code against their own ACs. ### The reconciliation-trigger gap â€” why this is this story's problem (see AC6)

The epic's AC4 says `ReconciliationEngine` runs "after all plugs for a period are processed," but nothing in Stories 6.1â€“6.3's actual architecture produces an "all plugs for a period are done" event â€” `ProcessImportFunction` is a blob-trigger Function invoked independently, once per uploaded file (one `plugId` per invocation), with no batching, no fan-in, and no cross-job coordination of any kind. Deferring the trigger design to a later story isn't viable â€” Story 6.6 (Import UI) is frontend-only and has no bearing on this backend gap, and Epic 7's `DecompositionEngine` computes a **user-selected, read-time** period on demand, which is a completely different mechanism serving a completely different purpose (on-screen decomposition breakdown vs. an import-time data-integrity guard) â€” it must not be conflated with this story's `ReconciliationEngine`. AC6 resolves this the smallest way that still satisfies FR-27: reconcile once per import, scoped to the plugId's own covered period, checking that flat-wide attribution for that specific date range never impossibly exceeds what the main meter measured.

### InterpolationEngine â€” worked example

Anchors at `2026-01-01 = 1.0` and `2026-01-05 = 5.0` (`span = 4`), with `2026-01-02..04` missing. Preceding 7-day average happens to be `2.5` (computed from real days `2025-12-26..2026-01-01`, however many of those 7 are actually present). Raw linear interpolation: day `01-02` (`step=1`) â†’ `1.0 + (5.0-1.0)*1/4 = 2.0`; day `01-03` (`step=2`) â†’ `3.0`; day `01-04` (`step=3`) â†’ `4.0`. After the `Math.Min(raw, 2.5)` cap: `01-02 â†’ 2.0` (uncapped, below cap), `01-03 â†’ 2.5` (capped), `01-04 â†’ 2.5` (capped). All three inserted rows get `IsInterpolated = true`. The `GapRange` recorded is `(2026-01-02, 2026-01-04)` â€” one entry per contiguous run, not one per missing day.

### ReconciliationEngine â€” why endpoint-interpolation was rejected in favor of duplicating `BuildDailySeries`

An earlier, simpler design considered computing `MainMeterTotal` as `InterpolatedMeterValueAt(periodEnd) âˆ’ InterpolatedMeterValueAt(periodStart)` (interpolate the cumulative reading value at each boundary, then subtract) â€” mathematically cleaner, but it would silently diverge from every other consumption figure this app already displays (`KpiCalculator`'s `DailyConsumption` points, spike detection, trend chart), which all derive from the per-day-bucketed, `Math.Max(0m, delta)`-clamped `BuildDailySeries` allocation. A meter-reset (downward `KwhValue` jump, a pre-existing known gap per `deferred-work.md`) would be silently ignored by endpoint interpolation but correctly clamped to zero by the day-series approach, matching what the KPI dashboard would show for the same span. Consistency with the already-displayed numbers matters more here than saving one duplicated method â€” reconciliation failures need to make sense next to the dashboard the user is already looking at.
