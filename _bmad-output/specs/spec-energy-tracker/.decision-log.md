## 2026-06-20 — Open questions resolved; companion authored

**smart-plug-formats.md authored** from sample files in `sample-data/eve/` and `sample-data/meross/`. Key findings:
- Eve Home: single-sheet Excel ("Gesamtverbrauch"), 3-row metadata header (device/room/home), then ~10-minute interval Wh readings in reverse chronological order. Device and room names embedded in file. Unit is Wh per interval — must sum and divide by 1000 for daily kWh.
- Meross: daily-aggregate CSV with BOM, tab+comma delimiter quirk (`field\t,value\t`), device name in filename only (no in-file metadata). Unit is kWh/day — no aggregation needed. Date format YYYY-MM-DD, forward chronological.
- Unified timeline contract specified: plug_id, device_name, date, kwh (float).

**OQ resolved — spike threshold (CAP-3):** Threshold is configurable by the user; default is 2× the 7-day rolling average. CAP-3 success criterion updated accordingly.

**OQ resolved — budget input placement (CAP-10):** Annual planned spend is a separate settings area from tariff configuration. It represents the user's personal power expense budget; mid-year forecast breaches serve as a trigger to review tariffs. CAP-10 intent and success updated to reflect dedicated budget settings screen.

## 2026-06-20 — Auth approach updated to OIDC / OAuth 2.0

**Decision:** Authentication and authorization are based on OpenID Connect and OAuth 2.0. The identity provider is swappable via configuration; Azure Entra ID is the initial provider (natural fit for the existing Azure hosting). This replaces the earlier assumption of credential-based auth.

CAP-5 intent and success updated to reflect the OIDC flow and the provider-swap requirement. Corresponding constraint added. Stale assumption about username/password auth removed.

## 2026-06-20 — Team review incorporated; major spec update

**Changes from party-mode team review (John, Sally, Winston, Amelia) + owner input:**

**CAP-4 (Tariff):** Clarified immutability model. Contract period prices are locked once active; future tariff changes (including provider switches) can be pre-entered. Historical meter readings and smart plug imports can be entered retroactively and are costed at the period-correct tariff.

**CAP-6 (Smart plug import):** Gap-filling removed — decomposition is only computed for periods with actual imported data; periods without data surface as explicitly unavailable (see CAP-9). Error handling added: three categorized user-facing error messages (unreadable, retry, service unavailable), errors logged internally. Retroactive imports (past periods) explicitly supported.

**CAP-7 (Flat structure):** Made conditional — only required for decomposition/smart plug import; meter entry and KPI dashboard work without it. Default room template added (living room, bedroom, kitchen, bathroom, hallway). Multi-flat support added: users can create and manage multiple flats; all data is flat-scoped.

**CAP-8 (Device registration):** Significantly expanded. Device now carries general metadata (type, manufacturer, model, optional purchase date). Energy consumption now supports two approaches: (a) EU energy label — class rating + stated annual kWh; (b) self-measured average — daily or weekly kWh entered by user.

**CAP-9 (Decomposition):** Updated to reflect period-scoped availability: ranges without smart plug data show "decomposition unavailable" with import prompt rather than zeros.

**CAP-10 (Insights):** Delivery mechanism specified — dedicated insights page, daily schedule at 02:00, manual trigger, progress feedback, insights shown as discovered.

**CAP-11 (Onboarding) — new:** First-use one-time setup captures flat name, estimated annual kWh (baseline for invoice deviation), and tariff details (required: base fee + €/kWh; optional: provider, contract start, contract duration 1/6/12/24 months). Gates access to main app. Editable from settings.

**Constraints:** Multi-tenancy model specified — tenant auto-provisioned per user-id; each user can track multiple flats; all data flat-scoped; no cross-tenant access. Infrastructure partially specified: Azure Blob Storage for uploads, blob-triggered Function for async processing, Storage Queue for messaging. Data store TBD (Azure SQL vs Cosmos DB).

**Non-goals:** "Multi-flat management UI" removed from non-goals (multi-flat now in scope). "Multi-user management UI" retained as non-goal.

**Open questions added:** Data store choice, CAP-9 rounding tolerance, gap-filling within a plug's covered range.

## 2026-06-20 — Gap-filling resolved; i18n added

**OQ resolved — CAP-6 gap-filling within covered range:** Mid-period gaps (individual missing days inside an otherwise covered export) are detected, user is notified with the affected date ranges, and gaps are filled by linear interpolation. Interpolated values are internally marked for tracking. In normal downstream processing (decomposition, insights) interpolated values are used transparently without further flagging. CAP-6 updated accordingly. Open question removed.

**New CAP-12 — Multi-language / locale support:** App launches with English and German. Language selection drives not only UI text but all number, date, and currency formatting (locale-aware rendering). Raw data stored and transmitted locale-neutrally (ISO 8601, decimal-point numbers). Architecture constraint added: no hardcoded locale-specific formatting anywhere in codebase.

## 2026-06-20 — Second party-mode review blockers resolved

**Multi-flat navigation (CAP-7):** Flat switcher component in app header; last active flat remembered across sessions. Flat create/delete in settings. Flat deletion is a full cascade delete of all associated data (readings, tariff entries, smart plug data, flat structure, devices). CAP-7 intent and success updated.

**CAP-11 annual kWh estimation:** Two input modes — specific numeric value, or household-size preset (1p=1500, 2p=2500, 3p=3500, 4p=4250 kWh). Field is not skippable; both modes cover the "I don't know exactly" case via presets. CAP-11 updated.

**Currency:** Follows user locale (€ for de-DE, $ for en-US). Stored as fixed-decimal (C# decimal equivalent) throughout data and API layers — no floating-point monetary values. CAP-12 and constraints updated.

**CAP-6/CAP-9 interpolation accounting:** Interpolated values count as attributed consumption. Accounting identity tolerance (±0.1 kWh) is relaxed when interpolation is present in the period. A hint is shown to the user whenever viewing a period containing interpolated data. CAP-6 and CAP-9 updated.

**CAP-10 scheduled job timezone:** UTC. All datetimes stored and transferred with explicit timezone offset (ISO 8601). Constraint added.

**New companion — locale-formats.md:** Canonical format spec for de-DE and en-US covering number separators, currency, date, and time formats. SPEC.md companions updated.
