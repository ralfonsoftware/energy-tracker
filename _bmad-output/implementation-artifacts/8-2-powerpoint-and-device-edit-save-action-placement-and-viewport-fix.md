# Story 8.2: PowerPoint & Device Edit — Save-Action Placement & Viewport Fix

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want the Save action for a new or edited Power Point to be visible right where I'm working, and the Cancel/Save buttons on the Device edit screen to always be visible without scrolling,
so that I don't have to navigate back to a distant part of the screen to find out whether my change was ever committed.

## Acceptance Criteria

1. **Given** the room-detail view (`RoomEditor.tsx`), where adding/editing a Power Point currently has no local save affordance — the only way to commit it today is navigating back to the room list and clicking either that room's own per-room Save button (Story 8.1) or the page-level "Speichern" button, **when** the user adds or edits a Power Point (name, Smart Plug ID) or a Device within it, **then** a sticky/fixed bottom Save action is visible within `RoomEditor.tsx` itself, without requiring the user to scroll or navigate away, so committing a change never requires leaving the room-detail view.

2. **Given** the new sticky Save action in `RoomEditor.tsx`, **when** it is clicked, **then** it reuses the exact same `handleSaveRoom` logic and `useUpdateFlatStructure(flatId)` mutation instance already established in `FlatStructureEditor.tsx` by Story 8.1 — no new mutation, no new `isPending`-like flag — so the room-list Save button (Story 8.1) and this new in-room Save button always agree on payload and pending-state.

3. **Given** a room's dirty-check today (Story 8.1's `isRoomDirty` inline expression) only compares the room's current name against its `originalName`, **when** this story is implemented, **then** the dirty-check is broadened into a shared function that also detects Power Point/Device changes (any difference between the room's current `draftRooms` content and its `lastSaved` wire-shape snapshot) — used consistently by **both** the room-list Save button and the new `RoomEditor`-local Save button, so a Power Point edit alone (no rename) correctly enables Save in both places, not just after a rename.

4. **Given** `handleSaveRoom`'s payload is already built from the room's live `draftRooms` content, not a stale snapshot (fixed in Story 8.1's code review — `toRoomInput(room, trimmedName)`), **when** this story adds `RoomEditor.tsx` as a second call site for the same handler, **then** no change to `handleSaveRoom`'s payload-building is needed — this AC exists to confirm the existing fix already covers the new call site correctly, and a regression test proves a Power Point edit made and saved from inside `RoomEditor.tsx` actually reaches the server payload.

5. **Given a gap found during story creation** — the existing flat-wide `hasBlankName(draftRooms)` and `findPlugIdConflict(draftRooms)` checks (used today only to gate the page-level Speichern button) scan **every** room's live draft, including rooms that are not even part of what a room-level save actually sends (per Story 8.1's AC9, a per-room save's payload reuses `lastSaved` — already-persisted, valid data — for every *other* room, never their live, possibly-invalid drafts), **when** the new `RoomEditor`-local Save button (or the existing room-list Save button) is evaluated for whether it should be enabled, **then** it must be gated by room-scoped checks that mirror exactly what will be sent: (a) a blank-name check scoped to *this room's own* Power Points only, and (b) a plugId-conflict check comparing this room's own Power Point plugIds against each other **and** against every *other* room's already-*saved* plugIds (from `lastSaved`) — never against another room's unsaved, live draft plugIds, which are not part of the payload and would produce a false-positive block. Using the existing flat-wide checks for this new gate would incorrectly block a valid, isolated room-level save because of unrelated invalid state sitting in a different room's live draft. If this room-scoped guard is ever bypassed by a race (e.g. a concurrent save elsewhere — already largely prevented by Story 8.1's shared `isPending` serialization), the existing generic `onError` → `editor.saveError` handling is sufficient; no new error differentiation is required.

6. **Given** `DeviceEditor.tsx`'s Cancel/Save button row, currently placed via in-flow layout (a `<div className="flex-1" />` spacer inside a `minHeight: 100vh` column) rather than pinned to the viewport, **when** the Device edit screen is opened in Safari (iOS or macOS), where the dynamic browser toolbar changes the effective visible viewport height, **then** the Cancel/Save row remains fully visible without requiring the user to scroll — implemented via a sticky/fixed-positioned button bar, not a `100vh`-in-flow assumption — verified by an actual manual check in Safari, not only Chrome/Firefox.

7. **Given a gap found during story creation** — `FlatStructureEditor.tsx` (and therefore `RoomEditor.tsx`/`DeviceEditor.tsx`, both rendered as its child views) is reached via `/settings/*` and rendered inside `AppShell.tsx`'s `<main>` element, which is the **actual scrolling ancestor** (`overflow-y-auto`) — not the document body/window — and on mobile viewports (`<md`) a `position: fixed` `BottomTabBar` (`client/src/components/BottomTabBar.tsx`, height `calc(72px + env(safe-area-inset-bottom, 0px))`) overlays the bottom of `<main>`'s own scrollable area, which is exactly why `<main>` already reserves `calc(84px + env(safe-area-inset-bottom, 0px))` of its own bottom padding (`AppShell.tsx`), **when** the new sticky/fixed button bars (AC1, AC6) are implemented, **then** they must use `position: sticky` (which naturally scopes to `<main>`'s scrollport and, unlike `position: fixed`, automatically excludes the desktop 200px sidebar without a media query) with `bottom: calc(84px + env(safe-area-inset-bottom, 0px))` on mobile and `bottom: 0` on desktop (`md:`) — reusing `AppShell.tsx`'s exact existing offset value rather than inventing a new one — so the new bar sits visibly above the `BottomTabBar` on mobile instead of colliding with or hiding behind it.

8. **Given** this story introduces the same sticky-bottom-bar positioning pattern in two places (`RoomEditor.tsx` per AC1, `DeviceEditor.tsx` per AC6), **when** it is implemented, **then** it is extracted into one shared, purely presentational component within the `flat-structure` feature slice, reused by both screens — not copy-pasted twice — so the pattern is genuinely "documented as the one to reuse for any other full-height edit screen in this feature area" per the epic's intent, and a future screen in this slice can reuse it without re-deriving the Safari/tab-bar-offset math.

9. **Given** the existing page-level "Speichern" button and the Story 8.1 room-list Save button, **when** this story is implemented, **then** both remain functionally unchanged in trigger location and click behavior — Speichern still submits the full `draftRooms` via `toUpdateRequest`; the room-list Save button's *enablement logic* becomes more accurate (AC3, AC5's broadened/room-scoped checks) but it is not moved, relabeled, or given new click behavior. This story does not remove or relocate either existing save action, and does not change `PowerPointEditor.tsx`'s per-keystroke write-into-`draftRooms` behavior or `DeviceEditor.tsx`'s own local commit-then-Cancel/Save dialog behavior — only where Save/Cancel visually sit and what triggers a network persist.

## Dev Notes

### This story's core technical insight: the room-level save infrastructure from Story 8.1 already does the hard part

Story 8.1's code review fixed `handleSaveRoom` to build its payload from the room's **live** `draftRooms` content (`toRoomInput(room, trimmedName)`), not a stale snapshot — this was fixed to solve a different problem (a room's own Save button silently dropping that room's own unsaved Power Point edits), but it means `handleSaveRoom` is *already* exactly the function this story needs for a Power-Point-aware in-room Save button. This story's real work is: (1) broadening what counts as "dirty" so a Power Point-only edit is detected at all (today's dirty-check is name-only), (2) adding a second call site for the same handler inside `RoomEditor.tsx`, and (3) the room-scoped validity guards in AC5 so this new, more-frequently-triggered save path doesn't round-trip a 400/422 for reasons unrelated to the actual edit. No new mutation, no new payload-building logic, no new `isPending` flag.

### Why the blank-name/plugId guards must be room-scoped, not the existing flat-wide ones

`hasBlankName(draftRooms)` and `findPlugIdConflict(draftRooms)` already exist and gate the page-level Speichern button — reusing them naively for the new in-room Save button would be the *wrong* level of scope: they scan every room's *live* draft, but a room-level save's payload only ever contains **this** room's live draft plus every *other* room's already-`lastSaved` (already-persisted, known-valid) data (Story 8.1 AC9). A blank Power Point being typed in some other, unrelated room right now would incorrectly block this room's valid save if the flat-wide checks were reused — the exact class of bug Story 8.1's AC9 was written to prevent, just one level down (guard logic, not payload-building this time). AC5's room-scoped `hasBlankNameInRoom`/`hasPlugIdConflictForRoomSave` mirror precisely what will be sent.

### `<main>` is the scroll container, not `window` — this changes what "sticky" and "fixed" actually mean here

`FlatStructureEditor.tsx` is reached via `/settings/*`, rendered inside `AppShell.tsx`'s `<main ref={mainRef} className="flex-1 overflow-y-auto ...">` (`client/src/components/AppShell.tsx:23-29`). `<main>`, not the document/window, is the scrolling ancestor. This matters for two reasons:
1. `position: sticky` resolves against the nearest scrolling ancestor — using `sticky` here automatically confines the button bar to `<main>`'s content column, which already excludes the desktop `SidebarNav` (a separate flex sibling, `w-[200px]`, not inside `<main>`). A naive `position: fixed; left: 0; right: 0` bar would instead span the full viewport width and visually sit on top of the sidebar on desktop — `sticky` avoids that problem for free.
2. On mobile (`<md`), `BottomTabBar` (`client/src/components/BottomTabBar.tsx`) is `position: fixed; bottom: 0` with `height: calc(72px + env(safe-area-inset-bottom, 0px))`, rendered as a sibling *outside* `<main>` — it visually overlays the bottom of `<main>`'s own scrollable area. This is exactly why `<main>` already reserves `pb-[calc(84px_+_env(safe-area-inset-bottom,0px))] md:pb-0` (`AppShell.tsx:25`). The new sticky button bar must reuse that *exact* offset (`bottom-[calc(84px+env(safe-area-inset-bottom,0px))] md:bottom-0`) so it sits visibly above `BottomTabBar` on mobile instead of rendering underneath/behind it. This constraint is not mentioned anywhere in the epic text — it only becomes apparent by reading `AppShell.tsx`/`BottomTabBar.tsx`/`router.tsx` together, which is why it's called out here explicitly rather than left for the dev agent to discover (or miss) during implementation.

### Why `position: sticky`, not `position: fixed`, satisfies the epic's literal wording

The epic AC says "implemented via a sticky/fixed-positioned button bar, not a `100vh`-in-flow assumption." `position: sticky` on a trailing element within an `overflow-y: auto` container is the standard technique for a bottom action bar that stays visible immediately on load (not just after scrolling) even when the container's content is taller than the visible viewport — this is what actually fixes the Safari dynamic-toolbar problem (the bug isn't really about `100vh` per se; it's that flow-positioned content assumes a fixed viewport height that Safari's toolbar makes untrue). `position: fixed` would also technically satisfy "always visible," but (per the point above) would require extra media-query logic to avoid overlapping the desktop sidebar, which `sticky` avoids by construction. Use `sticky`.

### Scope boundary: this story does not change how or when `draftRooms` gets locally mutated

`PowerPointEditor.tsx` still writes into `draftRooms` on every keystroke with no local commit gate (Story 8.1 Dev Notes) — this story does not add one; a half-typed Power Point name is still visible in `draftRooms` the instant the user types it, same as today. `DeviceEditor.tsx` still only commits into `draftRooms` when its own local Save is clicked (`onCancel` still discards cleanly) — this story only changes DeviceEditor's button *positioning*, not its commit semantics. What this story adds is a way to *persist* (network-save) the room's current `draftRooms` content from within `RoomEditor.tsx`, without navigating back to the room list or using the page-level Speichern.

### Backend is unmodified, again

`UpdateFlatStructureFunction.cs`, `UpdateFlatStructureValidator.cs`, `FlatStructureModels.cs` — all unmodified by this story. Confirmed (for AC5's guard design) that the backend's plugId-uniqueness check (`UpdateFlatStructureFunction.cs:75-82`) is flat-wide across the *entire submitted payload* and returns HTTP 422 with a generic detail message — the existing generic `onError → editor.saveError` handling already covers this if the client-side guard is ever bypassed by a race. `UpdateFlatStructureValidator.cs` confirms `Room.Name`/`PowerPoint.Name` are `NotEmpty()` (still the rule AC5's blank-name guard exists to pre-empt at the room-detail level, this time surfaced immediately in-context rather than only on the distant page-level Speichern).

Test naming: `FlatStructureEditor_Scenario_ExpectedBehavior`. `RoomEditor`/`PowerPointEditor` remain tested only through `FlatStructureEditor.test.tsx`'s integration-style navigation (no dedicated `RoomEditor.test.tsx` exists today — do not introduce one; the new logic in `RoomEditor.tsx` is thin/prop-driven by design, with the real decision logic living in `FlatStructureEditor.tsx` where it's already covered). `DeviceEditor.test.tsx` is the one existing dedicated test file in this feature slice — extend it only if `StickyActionBar`'s wrapping changes any of its 7 existing assertions (it should not).

- [x] [Review][Patch — Fixed] Save feedback state (`saveError`/`saveSuccess`) leaks across room navigation and can co-render within the same room [client/src/features/flat-structure/components/FlatStructureEditor.tsx (state ~L47-48, `setView` calls, `handleSaveRoom` ~L88-114), client/src/features/flat-structure/components/RoomEditor.tsx:93-108] — these flags are flat-editor-wide, never reset when `setView` changes which room is being viewed.
