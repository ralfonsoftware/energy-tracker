# Decision Log — energy-tracker Brief

## 2026-06-20

### Platform: Responsive web app first, native iOS later
**Decision:** Build responsive web app as v1. Native iOS (with widgets) is a confirmed follow-on.
**Reason:** User wants to explore native iOS options but recognises web app is the faster starting point. Mobile UX is the primary pain point driving the switch from spreadsheet.

### Auth: Required from day one
**Decision:** Authentication is in scope for v1.
**Reason:** App will be hosted on Azure and accessible from anywhere. Even for single-user use, securing access is non-negotiable. Architecture should support multi-user later without rework.

### Tariff model: €/kWh + fixed monthly fee with effective date — period-locked
**Decision:** Tariff has two components — variable (€/kWh) and fixed (€/month). Tariff entries are dated. Historical cost figures use the tariff that was active in each period (period-locked, not retroactively recalculated).
**Reason:** User reviews tariffs every 5–6 months. Period-locking preserves historical truth — a tariff change in the past was as it was, and seeing that change reflected in cost analysis is useful signal, not noise.

### Smart plug data: File import only (no API / hub)
**Decision:** v1 imports files only. Eve Home = Excel export. Meross = CSV export.
**Reason:** User already has these exports as part of current workflow. Direct API integration is future scope.

### Attribution model: Hybrid (smart plug + EU label + residual)
**Decision:** Three-tier attribution: smart plug data where available, EU energy label rated wattage for known-but-uninstrumented devices, explicit residual bucket for everything else.
**Reason:** Full smart plug coverage is not realistic (induction hob, etc.). The residual bucket must be visible, not hidden. EU label entry allows fridge and similar always-on appliances to contribute to the picture.

### Tech stack: .NET + Angular or React + Azure (Static Web App + Functions)
**Decision:** Backend in .NET Azure Functions, frontend as Azure Static Web App (Angular or React — TBD based on web criteria alone), hosted on Azure.
**Reason:** User knows .NET and Angular/React; has an Azure account. Static Web App + Functions is the most cost-effective model at low/zero traffic. iOS follow-on will be native Swift — no React Native or Capacitor path — so frontend framework choice is purely a web decision.
**Open question:** Angular vs React. iOS constraint removed; decision is now purely about build speed, bundle size, and developer preference for this web app.

### iOS follow-on: Native Swift, not cross-platform
**Decision:** The iOS app will be fully native Swift. Targets: iPhone primarily, iPad and Apple TV as exploration.
**Reason:** User explicitly wants to explore native iOS options — widgets, platform-native patterns. React Native / Capacitor / cross-platform bridges are not the goal.

### Release split: R1 = tracking core, R2 = decomposition layer
**Decision:** Split v1 into two releases. R1 delivers meter tracking, KPIs, tariff, and auth. R2 adds smart plug import, flat structure, device attribution, EU label entry, and the decomposition view.
**Reason:** User proposed the split. R1 is a usable replacement for the spreadsheet; R2 adds the intelligence layer. Cleaner delivery milestones.

### Smart plug export cadence: variable; gaps projected from trend
**Decision:** Exports will be uploaded at least monthly, targeting weekly. The app will fill gaps in smart plug coverage by projecting from the current consumption trend rather than leaving holes.
**Reason:** Real-world cadence is unpredictable. Projecting from trend is more useful than showing blank periods.

### Insights focus: Standby, replacement candidates, budget alerts
**Decision:** Three insight categories in Release 2: high standby draw, high-consumption replacement candidates, budget pressure signals.
**Reason:** These are the specific decisions the user wants to make. Not generic analytics — targeted actionable signals.
