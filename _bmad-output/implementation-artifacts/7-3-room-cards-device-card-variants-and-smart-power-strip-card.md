# Story 7.3: Room Cards, Device Card Variants & Smart Power Strip Card

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want to see my consumption grouped by room with device cards that visually distinguish between directly measured devices and estimated ones, and Smart Power Strip sub-devices that clearly show which are configured and which are not,
so that I understand both what I know precisely and where estimates are being used.

## Acceptance Criteria

1. **Given** `RoomCard.tsx` (currently a header-only placeholder from Story 7.2), **when** it renders for a room, **then** it also renders a vertical list of child device cards beneath the header; **and** in `DecompositionTab.tsx`, `data.rooms` is sorted by descending `kwh` (client-side — the backend orders rooms by `Room.SortOrder`, not kWh) before being mapped to `RoomCard`s.
2. **Given** a room's `devices` array, **when** `RoomCard` builds the child card list, **then** devices are partitioned into a "measured" group (`approach === 'Measured'` **or** `isSmartStrip === true`) and an "estimated" group (`approach === 'EuLabel' | 'SelfMeasured'`), each group sorted by descending `kwh`, measured group rendered first as **one interleaved list** (a `DeviceCard`-rich instance and a `SmartStripCard` instance within the measured group are sorted together by `kwh`, not rendered as two separate sub-lists); devices with `approach === 'None' && isSmartStrip === false` (zero contribution, per Story 7.1 AC2) are excluded from the list entirely — they render no card.
3. **Given** a device with `approach === 'Measured'` and `isSmartStrip === false`, **when** its `DeviceCard` renders, **then** it uses the **rich/measured variant**: larger card, device name prominent, kWh and cost values, a "Measured" badge, no estimate disclaimer.
4. **Given** a device with `approach === 'EuLabel'` or `'SelfMeasured'`, **when** its `DeviceCard` renders, **then** it uses the **compact/estimated variant**: smaller card, device name, kWh and cost values, an "Estimated" badge, and a one-line disclaimer: `t('deviceCard.disclaimerEuLabel')` ("Based on EU label") or `t('deviceCard.disclaimerSelfMeasured')` ("Based on self-measured average") matching `approach`.
5. **Given** a device with `isSmartStrip === true`, **when** `SmartStripCard.tsx` renders, **then** the card header shows the strip's `name`, total `kwh`, and `cost` (this total is measured, not estimated); sub-device rows from `subDevices` render beneath the header in a compact list, ordered configured-first then unconfigured, each sub-group sorted by descending `kwh`; if `subDevices` is an empty array (a Smart Power Strip PowerPoint with zero devices attached — a real, possible case, not an assumed invariant — see Dev Notes), the card renders header-only with no sub-device section and no error.
6. **Given** a sub-device row where `isConfigured === true`, **when** rendered, **then** it shows at full opacity (1.0), the sub-device `name`, its `kwh`/`cost` share, and no configure hint.
7. **Given** a sub-device row where `isUnconfigured === true`, **when** rendered, **then** it renders at `opacity: 0.45`, shows `name`, its equal-share `kwh`/`cost`, and an inline `t('smartStripCard.configureHint')` chip reading **"Go to settings"** — not "Configure": the destination is the structure list, not the specific PowerPoint, so the label must not promise more than it delivers (see Dev Notes); clicking the chip calls an `onConfigure` callback passed down from `DecompositionTab.tsx` (the same `onImport` prop-drilling pattern already used for `DecompositionUnavailable` — `DecompositionTab.tsx` already owns a `useNavigate` call and `DecompositionTab.test.tsx` already mocks `react-router-dom`'s `useNavigate`, so this reuses the existing pattern and its existing test mock verbatim), which navigates to `/settings/structure`.
8. **Given** a room where `devices` is empty, or every device has `approach === 'None' && isSmartStrip === false`, **when** `RoomCard` renders, **then** it renders one compact card reading `t('roomCard.directConsumption')` ("Direct consumption") with the room's `kwh` instead of the device list; no badge is shown.
9. **Given** device/room ordering and grouping logic, **when** implemented, **then** it lives in `RoomCard.tsx` (device grouping/sort, one interleaved measured-then-estimated list — see AC2) and `DecompositionTab.tsx` (room sort), each as a pure, module-level function — not inlined in JSX, not in the API layer, not a new shared util (consistent with this feature's established per-component logic pattern, Story 7.2 Dev Notes); the room-sort call site in `DecompositionTab.tsx` carries an inline comment noting the deliberate divergence from the API's `Room.SortOrder`.

## Dev Notes

- **Scope boundary honored from Story 7.2:** `RoomCard.tsx` was intentionally left as a header-only placeholder in 7.2 specifically for this story to extend — do not create a parallel/duplicate room card component.
- **No backend or API-client changes needed.** `DecompositionModels.cs` (`api/Features/Decomposition/DecompositionModels.cs`) and the frontend types already defined in `decompositionApi.ts` (`DeviceDecomposition`, `SubDeviceDecomposition`, `RoomDecomposition`) fully cover every field this story needs: `approach` ('Measured'|'EuLabel'|'SelfMeasured'|'None'), `isSmartStrip`, `subDevices` (nullable list), `isConfigured`/`isUnconfigured`. Enum values serialize as their literal C# member names (`JsonStringEnumConverter` wired into both `Http.Json.JsonOptions` and `Mvc.JsonOptions` in `Program.cs` — fixed in a prior story after a real production incident; do not re-litigate this).
- **Ordering is backend-silent — this story owns it entirely client-side.** `DecompositionEngine.cs` orders rooms by `Room.SortOrder` (not kWh) and has no ordering at all for `Devices`/`SubDevices` (insertion/PK order). AC1's "rooms ordered by descending kWh" and AC2's "Measured before Estimated, descending kWh within group" are not satisfied by the API response as-is — sort client-side per Tasks 3 and 4. Do not attempt to "fix" the backend ordering; that's out of scope and would touch a just-reviewed, tested engine (Story 7.1).
- **`isSmartStrip` implies a plugId already** — `DeviceDecomposition` has no `PlugId` field, but `DecompositionEngine.cs` only ever sets `IsSmartStrip: true` when the PowerPoint's `PlugId` is non-null. Treat `isSmartStrip === true` as sufficient for the "Measured" ordering group without looking for a plugId field that doesn't exist on this type.
- **Badge color tokens — resolved gap, revised after design/architecture review.** The epic's AC text names `accent-success`/`accent-warning` CSS tokens; neither exists anywhere in this codebase's design system. The initial draft of this story planned to reuse `accent-under-budget`/`accent-tariff-locked` directly; design and architecture review both independently flagged that as wrong — it sends a misleading signal (amber reads as a warning everywhere else in this app, but "Estimated" is a normal state for roughly half a user's devices, not a caution) and creates coincidental coupling across three unrelated domain concepts (budget status, tariff constraint, measurement provenance) that will evolve independently. Resolution: add two small new tokens to `client/src/index.css`'s `@theme` block — `--color-accent-success: var(--color-accent-under-budget);` (an alias today, its own identity going forward) and `--color-accent-neutral: rgba(255, 255, 255, 0.45);` (matches the actual UX mockup's dimmed "Estimated" badge color exactly, deliberately not amber/warning). Use `text-accent-success`/`text-accent-neutral` for the two badges — see Task 1.
- **`approach = 'None'` non-strip devices are a real gap the epic doesn't literally cover** — Story 7.1's AC2 states these contribute `Kwh = 0` but the epic's device-card ACs (7.3) only branch on Measured/EuLabel/SelfMeasured, leaving unaddressed what renders for a device that has neither a plug nor a configured estimation approach. Resolution (AC2/AC8 above): exclude them from the individual device list — a 0-kWh card with no badge would be pure noise — and only collapse the whole room to the "Direct consumption" fallback when **every** device in it is in this state (or the room has no devices at all). A room with a mix of real devices and None-approach devices simply omits the None ones from the visible list; nothing is lost since they contribute zero and the room total already reflects that. Design review flagged this as visually inconsistent with how the *same* "no measurement" state is handled on a Smart Power Strip sub-device (dimmed row + hint, not silent omission) — deliberately not changed here; giving standalone None-approach devices the same treatment is new UI surface (a new row state, new empty-copy), not a copy fix, and is tracked as a follow-up in `deferred-work.md` ("Consistent treatment for unmeasured devices — standalone vs. sub-device") rather than expanding this story's scope.
- **`isConfigured`/`isUnconfigured` are exhaustive and mutually exclusive by construction** — confirmed in `DecompositionEngine.cs:224-226`: `var isConfigured = d.ConsumptionApproach != ConsumptionApproach.None;` then `SubDeviceDecomposition(..., isConfigured, !isConfigured)`. No third state; a sub-device row will never silently fail to match either branch.
- **AC8's room-level fallback and AC2's device-level filter must share one definition, not two.** Both are driven by the same predicate — `approach === 'None' && !isSmartStrip` — applied at different levels: AC2 filters individual devices out of the render list; AC8's condition is `devices.length === 0 || devices.every(d => d.approach === 'None' && !d.isSmartStrip)`, i.e. "the AC2 filter would remove everything." Implement `RoomCard.tsx`'s fallback check by reusing the exact same predicate AC2's filter uses, not a second, independently-written condition, so the two can't drift out of sync.
- **The "Go to settings" chip's navigation target is intentionally shallow — and the label was chosen to say so.** `FlatStructureEditor.tsx` has no route-based or query-param-based way to deep-link to a specific PowerPoint — its internal `view` state (`{ type: 'device'; roomKey; powerPointKey; deviceKey }`) is keyed by client-generated `crypto.randomUUID()` draft keys that only exist after `useFlatStructure` data loads, not stable route-addressable IDs. Building true deep-linking would mean adding new state-plumbing to `FlatStructureEditor.tsx` — out of scope for a card-rendering story. This story navigates to `/settings/structure` (the existing route, landing on the room list) and stops there. Design review flagged the epic's literal "Configure" wording as over-promising a destination the app can't deliver, so the chip is labeled `t('smartStripCard.configureHint')` = "Go to settings" instead — an honest label for the actual destination. Deep-linking is tracked as its own follow-up in `deferred-work.md`, not a code comment that can evaporate.
- **Follow the existing `onImport`-style prop-drilling convention** for the "Go to settings" CTA (`DecompositionTab.tsx` owns `useNavigate`, passes a callback down through `RoomCard` → `SmartStripCard`) rather than importing `useNavigate` directly inside a leaf card component. This isn't just a stylistic match — `DecompositionTab.tsx` already imports `useNavigate` (for the existing Import CTA) and `DecompositionTab.test.tsx` already mocks `react-router-dom`'s `useNavigate`; routing this new callback through the same component reuses that existing mock rather than requiring a new one in `SmartStripCard.test.tsx` or `RoomCard.test.tsx`.
- **No shared `Badge` component exists** in this codebase (`CostGapBadge.tsx` is a bespoke, unrelated popover-trigger component) — write badge markup inline in `DeviceCard.tsx`/`SmartStripCard.tsx` as a styled `<span>`, consistent with this feature's established per-component duplication pattern (no shared formatter, no shared type file — see Story 7.2 Dev Notes) rather than introducing a new shared primitive for two use sites.
- **Testing pattern to follow** (established across all existing `decomposition/` test files): Vitest globals (no `describe`/`it`/`expect` imports), co-located `.test.tsx`, `vi.mock('react-i18next', () => ({ useTranslation: () => ({ t: (k) => k }) }))` with assertions checking for the raw i18n key text, `userEvent.setup()` for click interactions. Pure prop-driven components (`DeviceCard`, `SmartStripCard`) need no router/query wrapper or mocking, following `ResidualCard.test.tsx`'s minimal pattern — only `RoomCard.test.tsx` needs the `compareDocumentPosition`-based ordering-assertion pattern already used in `DecompositionTab.test.tsx`. Test naming: `ComponentName_Scenario_ExpectedBehavior`.
- **i18n:** `decomposition` namespace already registered in `client/src/lib/i18n.ts` — only add new keys to the two locale JSON files, no registry changes.
- **VSA/architecture compliance:** all new files live under `client/src/features/decomposition/components/`; no cross-slice imports; `flatId` is not needed by any component in this story (all data arrives already scoped via `DecompositionTab`'s existing `useDecomposition` call).
