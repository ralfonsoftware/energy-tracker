## What Went Well

1. **The "gap found during story creation" pattern caught real architecture gaps again, twice in a row.** Story 7.1's AC10 discovered that "sum of MainMeter daily readings" (the epic's literal wording) is physically meaningless for a cumulative meter counter, and reused `ReconciliationEngine`'s day-allocation algorithm instead — then *proved* no double-counting with a hand-derived partial-range test (62.5 kWh, worked out independently of the implementation) rather than just asserting it. AC11 caught that `TariffResolver.ResolveAsync` — the epic's named call target — has zero real callers anywhere in the codebase (confirmed by full-repo grep, re-confirmed again during this retro), and correctly redirected to `KpiCalculator`'s actual in-memory pattern instead of introducing an N+1 query with no precedent. AC12 discovered the data model has no "Smart Power Strip" entity at all and derived the concept structurally from `PlugId` + device count.
2. **"Prove it, don't assert it" gates showed up explicitly as ACs**, not just as review culture: Story 7.1's Task 5 required a literal `grep -n "TariffResolver"` returning zero matches before the task could be marked done; AC10's regression test was explicitly redesigned mid-review because the first version was "trivially correct regardless of internal allocation" and wouldn't have caught a real bug.
3. **Cross-functional design review caught two UX mistakes before they shipped**, not after. Sally flagged that the epic's literal `accent-success`/`accent-warning` badge tokens don't exist and that reusing the budget-warning amber for "Estimated" devices would misleadingly signal "something's wrong" for a normal state affecting half a user's devices — two new tokens were added instead. John caught that "Configure" over-promised a destination `FlatStructureEditor` can't currently deep-link to, and scoped the CTA down to an honest "Go to settings" label, pushing real deep-linking to a follow-up story rather than silently shipping a broken promise.
4. **Zero production incidents**, versus one in Epic 6 (the enum-serialization bug). All three Epic 6 retro prep items (deferred-work trigger process, cascade-delete helper extension for `ImportJobs`/`SmartPlugDailyData`/`SmartPlugIntervalData`, HTTP-level enum regression test) landed before Story 7.1 even started — second epic in a row where pre-epic commitments shipped as dedicated work ahead of time rather than squeezed into the first story.
5. **Ralf: "quite happy" overall** — the epic delivered what was scoped, with no functional surprises.

## Challenges & Patterns

1. **Zero manual live-Azure verification this epic**, breaking the streak Epic 6's retro specifically praised (Ralf's Story 6.6 walkthrough caught a UX bug — a broken device-association dropdown — that automated review structurally could not see). Story 7.1's own dev agent flagged this gap in its own Completion Notes for Story 7.3: *"Not verified in a live browser... Correctness relies on the unit/component test suite... flagging this explicitly per project verification standards."* Three new interactive components shipped (device cards, smart-strip sub-device rows with opacity states, a configure-hint chip) without a human ever clicking through them on the deployed app.
2. **A spec-formula problem shipped as a deferred item, contradicting the epic's own prose.** Story 7.1's review found that the Smart Power Strip "unconfigured remainder" formula (`device_share = (device_estimated_kWh / sumConfiguredEstimates) * stripMeasuredTotal`) makes `sum_of_configured_shares` always equal the full measured total in any strip with ≥1 configured device — so the "remainder" unconfigured devices are supposed to split equally is always ≈0. The code correctly implements the literal D-44 formula; the formula itself doesn't do what its own prose claims. Deferred as needing a `D-44` amendment, not a point fix.
3. **The "second-look review pass on any patch touching a non-negotiable rule" trial (carried since the Epic 4 retro) is now unexercised for a third straight epic**, despite what looks like a real qualifying case: Story 7.1's AC11 explicitly and deliberately overrode `project-context.md:423`'s stated invariant that `TariffResolver.ResolveAsync` is "the only correct path" for period-accurate costing. That's precisely the kind of patch the trial was meant to catch, and there's no evidence the second-look pass was triggered for it.
4. **Ralf's usage feedback surfaced a cross-epic pattern, not an Epic 7-specific bug**: flat structure saving locations are inconsistent and confusing, save-button placement is inconsistent across the app, and card layouts have drifted from his original vision. Winston's diagnosis: each epic's UI decisions get validated against that epic's own mockup, but nothing in the process checks new work against how *other* already-shipped features already do saves and layout. This is real, but by design not an "Epic 7 mistake" — it's accumulated debt across several epics that only a cross-feature audit (not epic-prose review) can catch.

## Significant Discovery — Roadmap Change

Epic 8 (Actionable Insights) is fully scoped and technically sound as written — nothing from Epic 7 invalidates its content. However, Ralf's usage feedback (recurring save-location/button-placement/card-layout inconsistency) is a real, cross-epic UX debt problem that he explicitly wants addressed as its **own dedicated epic**, and — per his decision during this retro — **before** Insights work begins.

**Recommended roadmap change:** Insert a new epic ("UI & Behavior Consistency Alignment" or similar) as **Epic 8**; renumber the current Epic 8 (Actionable Insights) to **Epic 9**. This mirrors the Epic 5.5 precedent (a story inserted mid-roadmap after the Epic 4 retro), except this is a full epic insertion, not a single story.

**This retrospective does not build that epic** — that requires `bmad-create-epics-and-stories` or `bmad-sprint-planning` to formally insert and scope it, seeded by Ralf's own cross-app walkthrough (not epic prose, per Winston's diagnosis above).
