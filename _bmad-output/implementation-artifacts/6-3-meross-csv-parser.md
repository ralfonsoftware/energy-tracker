# Story 6.3: Meross CSV Parser

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want to upload a Meross `.csv` export and have it parsed correctly, with zero-consumption days stored as valid data rather than treated as gaps,
so that my Meross plug timeline is accurate and days with no consumption are not incorrectly interpolated.

## Acceptance Criteria

1. **Given** a valid Meross `.csv` file (UTF-8 with optional BOM, tab-separated with per-value comma prefix, filename `Power Monitor Day Data - {device_name} - {YYYYMMDD}.csv`), **when** `MerossParser.cs` processes it, **then** device name is extracted from the filename using the specified pattern; UTF-8 BOM is stripped before parsing; trailing whitespace and empty rows are removed; each remaining row is parsed as `(Date: date, KwhValue: decimal)`.

2. **Given** a Meross row with `Power Consumption-(kWh)` value of `0.000`, **when** parsed, **then** it is stored in `SmartPlugDailyData` with `KwhValue = 0` and `IsInterpolated = false` — it is **not** treated as a missing date subject to gap detection or interpolation (FR-25).

3. **Given** the parsed Meross rows, **when** written to the database, **then** rows are upserted into `SmartPlugDailyData` only — no `SmartPlugIntervalData` rows are created for Meross imports; standby detection is not available for this format.

4. **Given** `MerossParserTests.cs` in `api.Tests/Features/SmartPlugImport/`, **when** run, **then** tests cover: valid file produces correct daily totals; UTF-8 BOM file parses correctly; zero-value row stored as valid data not flagged as a gap; device name extracted from filename; empty rows ignored.

5. **Given** a gap found during story creation — **no part of the pipeline built in Stories 6.1–6.2 ever captures or persists the original uploaded filename** (the blob path is `smart-plug-imports/{userId}/{flatId}/{importJobId}.{ext}` — extension only; `file.FileName` is read once in `UploadFunction` to extract the extension and then discarded), yet AC1 above requires the Meross device name to be parsed from the **filename**, not file content — **when** `UploadFunction.RunAsync` is extended to persist the uploaded file's original filename, **then**: `ImportJob` gains a required `OriginalFileName` (string) column (new EF Core migration, applied after `20260706132522_MakeSmartPlugIntervalDataTimestampUnique`); `UploadFunction` stores `Path.GetFileName(file.FileName)` on the created `ImportJob`; `ProcessImportFunction` reads `importJob.OriginalFileName` (already loaded from the DB row it fetches at the top of `RunAsync`) and passes it through to `MerossParser` for the `csv` dispatch branch only — no new blob-path segment or additional DB lookup is introduced, and `EveHomeParser`'s `xlsx` branch is untouched (it derives device name from cell A1, not the filename). See Dev Notes — "The OriginalFileName gap" for the full reasoning.

### The OriginalFileName gap — why this is this story's problem, not 6.6's

Discovered the same way Story 6.2 found the `PlugId` gap: by reading the *actual as-built* code of `UploadFunction.cs`/`ProcessImportFunction.cs` (as merged in Stories 6.1–6.2) against this story's own ACs, rather than trusting the epic text alone. `UploadFunction` reads `file.FileName` exactly once — to compute `ext` via `Path.GetExtension` — and then never touches it again; the blob it writes is named `{importJobId}{ext}` (extension only). By the time `ProcessImportFunction`'s blob trigger fires, the only string available for the file is the bare extension (`"csv"`, no leading dot, confirmed in Story 6.2's Dev Notes). Story 6.2 (Eve Home) never needed the original filename because its device name comes from cell A1 *inside* the file. This story's AC1, in contrast, requires the device name to come from the **filename** — a value that currently doesn't survive past `UploadFunction`. Story 6.6 (Import UI) is a frontend story with no bearing on this backend gap — deferring the fix there is not viable, exactly as Story 6.2 reasoned about the `PlugId` gap. Task 1 closes it with the smallest possible change: one new required column, populated from data `UploadFunction` already has in hand, threaded through the one dispatch branch that needs it.

### Why this parser must not blanket-wrap exceptions (critical — do not copy `EveHomeParser`'s pattern here)

`EveHomeParser.ParseFile` wraps *every* exception (`catch (Exception ex) { throw new UnreadableFileException(...); }`) as a deliberate simplification for `.xlsx` parsing. Copying that pattern verbatim into `MerossParser` looks like the obviously-consistent thing to do — it is not, and doing so silently breaks two pre-existing tests in `ProcessImportFunctionTests.cs`:

- `RunAsync_ServiceUnavailable_SetsFailedWithServiceUnavailable` feeds a `ThrowingStream(ImportServiceUnavailableException)` through the `csv` dispatch branch specifically *because*, until this story, `csv` routed to a dumb stub (`ConfirmBlobReadableAsync`) with no exception wrapping — the exception reaches `ProcessImportFunction`'s own `catch (ImportServiceUnavailableException ex)` block unmodified, correctly producing `ErrorCategory.ServiceUnavailable`.
- `RunAsync_UnhandledException_SetsFailedWithProcessingFailed` does the same with a plain `InvalidOperationException`, relying on `ProcessImportFunction`'s generic `catch (Exception ex)` → `ProcessingFailed`.

If `MerossParser` caught-and-rewrapped every exception the way `EveHomeParser` does, both of these would be reclassified as `DataUnreadable` instead — a regression these tests would catch, but only after the fact. The fix is a design choice, not an oversight to patch later: `MerossParser` should throw `UnreadableFileException` **only** for the one genuine content-format problem it can detect (an empty/blank file), and let every other exception — including a raw stream-read failure — propagate untouched. Per-row problems (a malformed line, an unparseable date or number) are handled as logged skips, never exceptions, so they never reach this question at all. This also means the pre-existing "garbage bytes" tests (`RunAsync_HappyPath_TransitionsPendingToComplete`, both `RunAsync_ConcurrencyConflictOn*` tests — all feed `new byte[] { 1, 2, 3 }` through the `csv` branch) continue to pass unchanged: those bytes decode to one non-empty "header" line with no `\n`, so `MerossParser` sees zero data rows and completes successfully with nothing to persist, exactly matching the old stub's behavior for that input.
