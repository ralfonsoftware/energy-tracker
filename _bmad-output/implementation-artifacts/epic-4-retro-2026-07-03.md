## Challenges & Learnings

### 1. A review fix round introduced its own bugs, caught only by a dedicated second round

Story 4.3 needed a genuine round-2 review — not re-verifying subtlety in the original implementation, but auditing round 1's *own patches*. Round 2 found 6 new issues introduced or left incomplete by round 1's fixes, including a dropped `PropertyNameCaseInsensitive` option (a documented non-negotiable backend rule) in a rewritten JSON-parsing path, and a submit-guard gap opened up *between* two sequential PATCH calls that round 1's own fix had introduced. This worked because a dedicated verification pass happened to run — not because it was a designed step.

### 2. A documented fix pattern was both correctly propagated and violated in the same story

Story 4.2's review fixed a UTC-vs-local-date bug in `TariffForm.todayIsoDate()`. Story 4.3 explicitly copied that fix pattern into the new `TariffLockIndicator` (good propagation). But Story 4.3's own new `formatDate` function, written days later in the same story, reintroduced the identical UTC bug — caught by review, not by self-review. Unlike the zero-data-state checklist item (a clean yes/no check), "don't use UTC for civil-date math" is scattered across every date-touching file and isn't a single checklist line.

### 3. Live testing during this retrospective found a deeper, unaddressed modeling gap

Testing on real hardware (Safari/iOS) during the readiness check surfaced that `EffectiveDate` (drives cost-period resolution, immutable after creation, defaults to "today" on the create form) and `ContractStartDate` (drives only the lock indicator, no cost-calculation effect) don't match the mental model of someone entering a real pre-existing contract. A tariff backdated via `ContractStartDate` alone, with `EffectiveDate` left at its default, shows "0 of N days covered" on the dashboard — and there is currently no way to correct a wrong `EffectiveDate` after creation (immutable via PATCH, no DELETE endpoint for tariffs). See Significant Discovery below — this is being routed to `bmad-correct-course`, not patched inline.

### 4. The Epic 3 action item to backfill `CompleteOnboardingFunction`/`PatchFlatFunction` test coverage slipped

Phrased as "consider backfilling" in the Epic 3 retro — soft language, and it slipped. Both functions still have only narrow validator-level tests, no dedicated HTTP-level Function tests, one epic later. Re-committed this retro with harder language (see Action Items).

## Live Testing Findings (surfaced during this retrospective's readiness check)

Ralf tested the shipped Epic 4 UI on Safari/iOS during the readiness discussion and found:

1. **[Significant]** Tariff cost-period coverage shows "0 of N days covered" for a real, currently-valid contract entered with a backdated `ContractStartDate`. Root cause: `EffectiveDate` (not `ContractStartDate`) drives `TariffResolver`'s cost-period lookup, defaults to "today" on the create form, and is immutable/undeletable after creation. Ralf's expectation — `ContractStartDate` should be the field that matters for cost calculation, with `ContractDurationMonths` becoming an advisory renewal hint rather than a locking precondition — requires a data-model discussion, not a point fix. **Routed to `bmad-correct-course`.**
2. **[Minor]** The tariff edit sheet's close ("✕") affordance is present but hard to notice/reach — discoverability issue, not a functional bug. Deferred to Epic 5 as a UI polish fix.
3. **[Minor]** Dashboard bottom tab bar is slightly overlaid by content on Safari/iOS; a related scroll/layout quirk appears on tab-switch. Deferred to Epic 5 as a bug fix (likely a missing `env(safe-area-inset-bottom)` accommodation).

- No optimistic concurrency control on `Tariff` PATCH — consistent with the rest of the codebase.
